# dust-night-registration

## 1. DUST is not freely transferable

There is no `sendDust(to, amount)` or `transferDust` primitive in the Midnight SDK or ledger. DUST is generated by NIGHT UTXOs. The only cross-wallet movement mechanism is the optional `dustReceiverAddress` parameter on `registerNightUtxosForDustGeneration`. A wallet registering its own NIGHT UTXOs can point the resulting generated DUST stream at a different wallet's DUST address. That's the only "send" pathway, and the originating wallet forfeits the stream for itself.

Lace's Send UI rejects DUST addresses as "invalid" destinations. That's correct behaviour. DUST addresses are generation receivers, not send destinations.

## 2. The registration call

Phase by phase:

1. Guard wallet is initialized and synced.
2. Enumerate available NIGHT UTXOs.
3. Estimate registration fee via `wallet.estimateRegistration(selected)`.
4. Build the recipe via `wallet.registerNightUtxosForDustGeneration(selected, nightVerifyingKey, signCallback, dustReceiverAddress)`.
5. proveTx → balanceTx → submitTx (the same three-phase sequence as any other Midnight tx).

Minimal direct call shape:

```typescript
import { createKeystore } from '@midnight-ntwrk/wallet-sdk-unshielded-wallet';

// nightVerifyingKey + signDustRegistration come from the unshielded keystore
// derived from your seed; do NOT re-derive from seed here, reuse the cached one.
const keystore = createKeystore(unshieldedSecretKey, networkId);
const nightVerifyingKey = keystore.getPublicKey();
const signDustRegistration = (payload: Uint8Array) => keystore.signData(payload);

// Pass undefined for dustReceiverAddress to default to the wallet's own
// DUST address. Pass a parsed bech32m DustAddress to redirect.
const recipe = await wallet.registerNightUtxosForDustGeneration(
  selectedUtxos,
  nightVerifyingKey,
  signDustRegistration,
  undefined,
);

// Standard 3-phase tx pipeline.
const provenTx = await providers.proofProvider.proveTx(recipe.transaction);
const balancedTx = await providers.walletProvider.balanceTx(provenTx);
const txId = await providers.midnightProvider.submitTx(balancedTx);
```

## 3. The bootstrap chicken-and-egg

`registerNightUtxosForDustGeneration` is a real on-chain transaction. Like any Midnight tx, it must be balanced, which requires DUST to pay the fee. A wallet with NIGHT and zero DUST cannot self-bootstrap directly through the SDK alone; balanceTx will throw with `insufficient funds` or `no dust UTXOs`.

Surface this as a distinct error class (e.g. `InsufficientDustForBootstrapError`) so the operator gets an actionable message instead of a generic `TxSubmissionError`.

### Working bootstrap

Open the same seed/mnemonic in Lace wallet (preview), navigate to the Midnight tab, and click through the tNIGHT Designation flow. Lace successfully runs `registerNightUtxosForDustGeneration` even on a wallet with 0 tDUST. This implies either a protocol-level first-tx subsidy or a Lace-side relayer; the operational fact is that it works every time.

After Lace designation, the backend (sharing the seed) inherits the on-chain designated state and starts accruing DUST automatically. First dust appears in ~90 seconds.

### Fallback: someone else delegates to your address

If you cannot run Lace (headless server, no GUI), a second party with existing funded NIGHT can run:

```typescript
await otherWallet.registerNightUtxosForDustGeneration(
  otherWalletUtxos,
  otherNightVerifyingKey,
  otherSignDustRegistration,
  yourDustAddressParsed, // <- redirected to you
);
```

They forfeit the stream for themselves until they re-register without the redirect.

## 4. The accrual timeline

After registration is finalized:

- 0 to 90s: DUST balance may still read 0 because the backend's wallet state hasn't refreshed against the new designation. `readWalletStateOnce` forces a refresh.
- ~90s: first non-zero DUST balance visible.
- ~167h to fill the tank: tank holds up to 5K tDUST and refills linearly while NIGHT remains designated. If you spend DUST, the tank refills from zero up to 5K over ~167h again.

Devs sometimes panic at 0 DUST in the first hour. Below ~10 minutes, that's expected. Above ~30 minutes with a synced wallet, designation likely failed silently. Check the tx receipt.

## 5. Lace UI bootstrap

For backends that share a seed with a Lace wallet on the same network, the recommended bootstrap is:

1. Import the seed/mnemonic into Lace.
2. Wait for Lace to sync (preview: ~5 min cold; preprod: longer).
3. On Lace's Midnight tab, click tNIGHT Designation once.
4. Wait ~90s for the backend to pick up the designated state via its own sync.
5. Verify via the sidecar: `GET /wallet/diagnostics` returns a non-zero `dustBalance`.

This avoids the headless bootstrap problem entirely.

## 6. Common failure modes

### "Backend wallet has NIGHT but DUST balance is 0"

In order of likelihood:

1. Designation never ran. Check on-chain: did `registerNightUtxosForDustGeneration` ever succeed for this wallet? If not, run it (Section 5) or call your adapter's equivalent of `designateDust()`.
2. Designation ran <90s ago. Wait, then re-read balance.
3. Backend reading the wrong DUST address. Three-address gotcha: `wallet.getWalletAddress()` returns the shielded coinPublicKey, not the DUST address. See `references/three-addresses.md`.

### "Tx fails with 'no dust UTXOs' / 'insufficient dust'"

In order of likelihood:

1. Wallet not fully synced. balanceTx looks at the wallet's local view of UTXOs; a partially-synced wallet may not yet see the DUST that accrued on chain. Check `appliedId >= highestTransactionId` via the sidecar diagnostics.
2. Designation never ran. Same as the previous mode.
3. DUST tank empty (you spent it). Check current balance; if very low, wait for refill (~167h to full from zero).

### "Lace shows DUST, but my backend reads 0"

The seed in Lace and the seed in the backend produce different DUST addresses. Either:

- Seed normalization differs (BIP39 PBKDF2 64-byte vs entropy-32-byte vs first-32-of-64). The format that matches Lace is `pbkdf2-full-64B`.
- HD account/role index differs. Lace and the canonical backend pattern both use `selectAccount(0).selectRoles([Zswap, NightExternal, Dust]).deriveKeysAt(0)`.

Run `scripts/address-derive.mjs` to verify:

```bash
node scripts/address-derive.mjs --mnemonic "<24 words>"
```

The script prints all three addresses derived from the seed. Compare against what Lace shows.

## 7. The seed-funding bootstrap recipe

For a fresh wallet starting from zero on `preview`:

```bash
# 1. Generate or import a 24-word mnemonic. Lace can do this.

# 2. Get the unshielded bech32m address (NOT the shielded hex).
node scripts/address-derive.mjs --mnemonic "<24 words>"
# Look for `unshieldedAddress: mn_addr_preview1...`

# 3. Drop NIGHT to that address via the preview faucet:
#    https://faucet.preview.midnight.network/

# 4. Wait for the faucet drop to land (~30s on preview).

# 5. Open the same mnemonic in Lace, click tNIGHT Designation.

# 6. Wait ~90s. Verify via sidecar:
curl -s -H "x-midnight-sidecar-token: $MIDNIGHT_SIDECAR_TOKEN" \
  http://127.0.0.1:8090/wallet/diagnostics | jq .balances
# Expected: { "night": "1000000000", "dust": "37000000", ... }

# 7. Now any tx the backend submits will balance correctly.
```
