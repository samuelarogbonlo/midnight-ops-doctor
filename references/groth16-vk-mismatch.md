# Groth16 verification-key mismatch: incident playbook

## What just happened

A Groth16 verifier contract bakes the verification key (vk) into its bytecode as immutable constants. The vk has 14 fixed scalars (G1/G2 points: `α`, `β`, `γ`, `δ`) plus a variable-length `IC[]` array (one G1 point per public input + 1). The contract uses these constants in its `verifyProof()` precompiled-pairing math.

If even one byte of the on-chain vk differs from the vk derived from your local `.zkey`, every proof fails verification. Even valid ones generated by your own circuit. The contract returns `false`. Functions gated on the verifier revert. Funds in any HTLC, escrow, or vault that depends on this verifier are stuck until timelock refunds fire.

There's no graceful degradation. There's no warning. The math is hard-deterministic.

## Why it happens

Five causes, ranked by frequency in real incidents:

1. **Deployed from a different zkey than expected.** Two `.zkey` files exist in the project (e.g. `circuit_0001.zkey` and `circuit_final.zkey`). The verifier was generated from one but the prover uses the other. Most common cause.
2. **`snarkjs zkey contribute` was run AFTER deploy.** Each contribution randomizes the `delta` G2 point. If you contribute on top of the canonical zkey, the local vk now differs from on-chain. Never run `zkey contribute` on a deployed zkey.
3. **CI/Docker regenerates the zkey on every build.** Multi-stage builds that include `circom` and `snarkjs` in a build stage produce a fresh vk on every cold rebuild (a fresh `delta` is sampled). Cache invalidation means silent vk drift, which means the next deploy is a vk-mismatch incident waiting for its trigger.
4. **snarkjs version drift between vk export and deploy.** The Solidity verifier template changed across snarkjs versions. Generating the verifier with `snarkjs@0.7.x` and exporting the vk JSON with `snarkjs@0.6.x` gives subtly different layouts.
5. **Verifier source compiled from one zkey but bytecode contains another.** Mostly happens when copy-pasting a Verifier.sol from a tutorial without regenerating from your own zkey.

## Detection: how Check 5 works

The verifier extracts vk both ways and byte-compares.

Local side:

```bash
snarkjs zkey export verificationkey ./build/settlement_proof_final.zkey ./local-vk.json
```

This emits a JSON with `vk_alpha_1`, `vk_beta_2`, `vk_gamma_2`, `vk_delta_2`, and `IC[]`. Note the snarkjs G2 element ordering convention: `[[x2, x1], [y2, y1]]` (the second-coefficient pair first; this is a snarkjs quirk that has tripped multiple verifiers).

On-chain side:

```bash
curl -X POST -H 'Content-Type: application/json' \
  --data '{"jsonrpc":"2.0","method":"eth_getCode","params":["0xYourVerifierAddress","latest"],"id":1}' \
  https://your-evm-rpc.example.org/rpc
```

The bytecode contains each scalar as a `PUSH(N) <bytes>` immediate. The script searches for each local-vk scalar inside the deployed bytecode using its `PUSH(N)` opcode prefix.

One subtlety: BN254 scalars with leading-zero bytes are emitted as shorter PUSH opcodes (e.g. a scalar with one leading zero byte uses `PUSH31` not `PUSH32`). The verifier accounts for all PUSH widths from `PUSH4` to `PUSH32`. Anything shorter is a guard against false positives; `PUSH1 0x00` (`6000`) appears in nearly every contract for reasons unrelated to vk.

If the script finds all local scalars in the deployed bytecode, vk matches. If any scalar is missing or appears under a different opcode, the script reports FAIL with the specific scalar name (e.g. `vk_gamma_2.x1`).

## Incident response playbook

When Check 5 reports FAIL, treat it as a P0 incident. Step through this in order. Don't skip ahead.

### Step 0 (T+0 min): Stop the bleeding

If your protocol is live and accepts user funds:

- Pause new HTLC creation if your contract has a pause function. Existing HTLCs can still refund via timelock; new ones would be unrecoverable until you redeploy.
- Notify your monitoring channel. Even if no funds are at risk yet, this incident is hours away from being one.
- Snapshot the deploy artifacts: `cp -r build/ build-incident-$(date +%s)/`. Don't let CI overwrite the local zkey while you investigate.

### Step 1 (T+5 min): Identify the canonical zkey

You have a deployed verifier. You have a local zkey. One of them is wrong. Identify which.

- Check your deploy commit in git. Was the verifier generated from a zkey checked-in at that commit?
- Re-run `snarkjs zkey export verificationkey <committed-zkey> /tmp/canonical-vk.json` against every zkey in your repo's history. Compare to the on-chain vk extracted by the verifier script.
- If one matches, that zkey is canonical.
- If none match, someone ran `zkey contribute` post-deploy, OR the deploy used a zkey that no longer exists. Most concerning case.

### Step 2 (T+15 min): Choose remediation

Remediation A: redeploy the verifier (most common path)

```bash
# 1. Generate new Verifier.sol from your local canonical zkey
snarkjs zkey export solidityverifier ./build/settlement_proof_final.zkey ./contracts/SettlementVerifier.sol

# 2. Deploy
npx hardhat deploy --network arbitrumSepolia --tags SettlementVerifier
# OR your deployment tool of choice

# 3. Update any contracts that reference the old verifier
# (example: IntentValidator.setSettlementVerifier(newAddress))

# 4. Re-run the deploy verifier
node scripts/deploy-verifier.mjs deploy-manifest.json
# Expect: Check 5 PASS
```

Use this when your local zkey is canonical and the deployed verifier isn't.

Remediation B: restore canonical zkey from ceremony output

Use this when the deployed verifier is canonical (e.g. you can't redeploy because it's mainnet) and your local zkey drifted.

```bash
# 1. Locate the original ceremony output. This SHOULD be checked into
# git (see "Prevention" below).
ls -la .git/lfs/objects/<ceremony-output-hash>

# 2. Replace local zkey
cp <canonical-zkey> ./build/settlement_proof_final.zkey

# 3. Verify
node scripts/deploy-verifier.mjs deploy-manifest.json
```

Never run `snarkjs zkey contribute` to "fix" the local zkey. Every contribution randomizes the delta point. You'd just make the mismatch worse.

### Step 3 (T+30 min): Communicate to users

If any user funds were locked between when the bad verifier deployed and when you fixed it:

- They cannot withdraw via the proof path. They can only refund via the HTLC timelock (typically 15-45 minutes).
- Communicate this clearly: "Affected addresses can refund via the timelock starting at HH:MM."
- Compensate gas costs for refund transactions if the bug was on your side.

### Step 4 (T+60 min): Post-mortem

Capture in your incident-response repo:
- The two divergent vk hashes
- The git commit of the bad zkey
- The CI step (if any) that regenerated it
- The control gap that allowed the bad verifier to deploy without Check 5 running first

## Prevention

Non-negotiable for any project shipping a Groth16 verifier:

1. Commit zkey artifacts to git. They're large but version-stable. The `.gitignore` should have explicit `!build/*.zkey` un-ignore rules. Trade-off: ~50MB per zkey, but the alternative is silent CI rebuilds that drift the vk.

2. Single-source-of-truth Dockerfile. The build stage that consumes the zkey should `COPY --from=context ./build/*.zkey ./build/`. Don't run `snarkjs zkey new` or `circom` inside the image. Use a slim runtime base + `COPY` only.

3. Run `deploy-verifier.mjs` as a CI gate. Any PR that touches the verifier contract or the zkey must pass Check 5 against staging before merging. Gate the prod deploy on Check 5 against prod RPC.

4. Pin snarkjs version exactly: `"snarkjs": "0.7.4"`, not `"^0.7.4"`. The caret is a footgun.

5. Never contribute to a deployed zkey. Once a zkey is the source of truth for a deployed verifier, it's frozen. If you need to extend the circuit, generate a new ceremony, deploy a new verifier, and migrate.

6. Document the canonical address per network in your project's deploy registry. Maintain a single source of truth (a registry doc, a JSON file, or your project's MEMORY/ADR equivalent) listing the current vk-aligned verifier address per chain. Anything not in the registry is retired, and `deploy-verifier.mjs` (or your equivalent CI gate) should refuse to validate against it.
